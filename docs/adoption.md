# Adoption & Quickstart

How to use the AI Workflow System in a new or existing project, with all
references resolving correctly.

---

## How adoption works

The framework is adopted by **copying its reusable files into your project,
preserving the same folder layout.** Because the layout is preserved, every
internal reference (`prompts/full-agent-workflow.md`, `docs/task-schema.md`,
`agents/developer.md`, the `.claude` wrappers) resolves at your project root
exactly as it does in this repository — nothing needs to be rewritten.

There are two kinds of files:

| Kind | Files | You edit them? |
|------|-------|----------------|
| **Framework files** (copied as-is) | `.claude/`, `agents/`, `prompts/`, `checklists/`, `docs/task-schema.md` | No — treat as upstream; re-copy to update |
| **Project working docs** (generated) | `CLAUDE.md`, `README.md`, `roadmap.md`, `architecture.md`, `decisions.md`, `qa.md`, `tasks.md` | Yes — these are your project's living docs |

The working docs are **generated by `/initialize-project`** (shaped after
`templates/project-template/`), not copied raw — so they are tailored to your
project. The template folder is the reference shape, not something you copy.

---

## What gets copied into a project

Copy these framework files/folders from this repo into your project root:

- `.claude/` — subagents + slash commands (Claude Code loads these automatically)
- `agents/` — the six role definitions the subagents and workflow reference
- `prompts/` — the canonical workflow and the command prompts
- `checklists/` — quality checklists
- `docs/task-schema.md` — the canonical task schema (into a `docs/` folder)

You do **not** copy `templates/`, `examples/`, `improvement-plan.md`,
`final-architecture-review.md`, or this `docs/adoption.md` — those are framework
documentation, not runtime files.

## Quickstart

### 1. Copy the framework files into your project

**Option A — installer script (Windows, recommended)**

From the framework repo, run:

```powershell
.\scripts\install-ai-workflow.ps1 -TargetPath C:\path\to\your-project
```

It copies only the runtime files, **never overwrites existing files** (re-run
with `-Force` to overwrite), and prints exactly what it copied and skipped. Then
continue to step 2.

**Option B — manual copy**

**PowerShell**

```powershell
$src = "C:\AI-Workflow-System"          # this repo
$dst = "C:\path\to\your-project"        # your project
Copy-Item "$src\.claude"     "$dst\.claude"     -Recurse
Copy-Item "$src\agents"      "$dst\agents"      -Recurse
Copy-Item "$src\prompts"     "$dst\prompts"     -Recurse
Copy-Item "$src\checklists"  "$dst\checklists"  -Recurse
New-Item -ItemType Directory -Force "$dst\docs" | Out-Null
Copy-Item "$src\docs\task-schema.md" "$dst\docs\task-schema.md"
```

**bash**

```bash
SRC=/path/to/AI-Workflow-System         # this repo
DST=/path/to/your-project               # your project
cp -r "$SRC/.claude"    "$DST/.claude"
cp -r "$SRC/agents"     "$DST/agents"
cp -r "$SRC/prompts"    "$DST/prompts"
cp -r "$SRC/checklists" "$DST/checklists"
mkdir -p "$DST/docs"
cp "$SRC/docs/task-schema.md" "$DST/docs/task-schema.md"
```

### 2. Generate the project working docs

Open your project in Claude Code and run:

```
/initialize-project
```

This detects your toolchain and creates/updates `CLAUDE.md`, `roadmap.md`,
`architecture.md`, `decisions.md`, `qa.md`, `tasks.md` (and `README.md` if you
don't already have one), filling in real commands. It will not delete your
existing documentation.

### 3. Work the backlog

```
/next-task        # select and implement the next approved task
# or
/workflow         # run the full end-to-end delivery workflow
```

Tasks follow `docs/task-schema.md`; the delivery process follows
`prompts/full-agent-workflow.md`.

---

## Verifying references resolve

After step 1, these paths should exist in your project and match every
reference used by the framework:

- `prompts/full-agent-workflow.md`
- `docs/task-schema.md`
- `agents/` (six role files)
- `.claude/agents/` and `.claude/commands/`

If a slash command or subagent says "follow `prompts/...`" or
"see `docs/task-schema.md`" and the file is present at that path, references
resolve. (See `examples/sample-project` in this repo for what an initialized
project's docs look like.)

---

## Updating the framework later

The framework files are upstream. To pick up improvements, re-copy the framework
folders (step 1) over your project's copies. Because you only edit the **working
docs**, re-copying framework files is safe and won't lose your project content.

---

## Optional: version control

If your project is a git repository, commit the framework files and the
generated working docs together. The workflow's final phase commits each
completed task; `initialize-project` will not commit unless you ask.
